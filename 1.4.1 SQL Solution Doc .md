# Org Chart Overhaul - SQL Solution  
**Recursive CTE Approach (MariaDB 10.4 / MySQL 8+)**

---

## 🎯 Objective

Transform a flat employee–manager table into an analytics-ready organizational model using SQL.

The solution generates:

- **Reporting Hierarchy** (Top → Employee)
- **Direct Reports** (Immediate reports)
- **Total Reports** (Direct + Indirect descendants)

This implementation uses a **Recursive Common Table Expression (CTE)** to model hierarchical data.

---

## 🧾 Input Data

Table: `employees`

| Column | Description |
|----------|------------|
| employee_name | Employee identifier |
| manager_name  | Direct manager (NULL for top-level employees) |

Assumption:  
Each employee has at most one manager (tree / forest structure).

---

# 🏗 Solution Architecture

The query is structured into four logical stages:

1. Data Normalization  
2. Recursive Hierarchy Construction  
3. Direct Reports Aggregation  
4. Total Reports Calculation  

---

# 1️⃣ Data Normalization

## Purpose

Ensure clean hierarchical relationships before recursion.

### Steps Performed

- Trim whitespace
- Convert empty manager values to `NULL`
- Remove invalid rows
- Remove accidental CSV header rows


UPDATE employees
SET employee_name = TRIM(employee_name),
    manager_name  = NULLIF(TRIM(manager_name), '');

DELETE FROM employees
WHERE employee_name IS NULL OR employee_name = '';

DELETE FROM employees
WHERE employee_name = 'Employee Name'
  AND manager_name = 'Manager Name';

Clean data is essential because recursive joins depend on exact string matches.
---
2️⃣ Recursive Hierarchy Construction
🎯 Purpose

Build the full chain of command using a recursive CTE.

Core Concept

A recursive CTE has two parts:

🔹 Anchor (Root Nodes)

Start from top-level employees:

SELECT
    employee_name,
    manager_name,
    employee_name AS reporting_hierarchy,
    0 AS lvl
FROM employees
WHERE manager_name IS NULL

Each root initializes its own hierarchy path.

🔹 Recursive Step

Attach employees whose manager already exists in the CTE:

SELECT
    e.employee_name,
    e.manager_name,
    CONCAT(h.reporting_hierarchy, ' > ', e.employee_name),
    h.lvl + 1
FROM employees e
JOIN heir h
  ON e.manager_name = h.employee_name
WHERE h.lvl < 100
🛡 Recursion Depth Guard
WHERE h.lvl < 100

Prevents infinite loops in case of circular references.
---
3️⃣ Direct Reports
🎯 Purpose

Count employees reporting directly to each manager.

direct AS (
    SELECT
        manager_name AS employee_name,
        COUNT(*) AS direct_reports
    FROM employees
    WHERE manager_name IS NOT NULL
    GROUP BY manager_name
)
Logic

Each row contributes +1 to its manager.

LEFT JOIN is later used to preserve employees with zero direct reports.
---
4️⃣ Consolidation (Hierarchy + Direct Reports)
hier_direct AS (
    SELECT
        h.employee_name,
        h.manager_name,
        h.reporting_hierarchy,
        COALESCE(d.direct_reports, 0) AS direct_reports
    FROM heir h
    LEFT JOIN direct d
      ON h.employee_name = d.employee_name
)

Ensures:

Individual contributors remain in output

Null counts are converted to 0
---
5️⃣ Total Reports (Direct + Indirect)
🎯 Purpose

Count all descendants under each employee.

Key Insight

If employee m is an ancestor of employee e, then:

e.reporting_hierarchy LIKE m.reporting_hierarchy + ' > %'

This string-based approach allows descendant detection.

Final Query
SELECT
    m.employee_name,
    m.manager_name,
    m.reporting_hierarchy,
    m.direct_reports,
    COUNT(e.employee_name) AS total_reports
FROM hier_direct m
LEFT JOIN hier_direct e
  ON e.reporting_hierarchy LIKE CONCAT(m.reporting_hierarchy, ' > %')
GROUP BY
    m.employee_name,
    m.manager_name,
    m.reporting_hierarchy,
    m.direct_reports
ORDER BY total_reports DESC, m.employee_name;
